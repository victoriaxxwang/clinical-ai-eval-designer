- **July 9 — TIER B (4) shipped: MeSH normalization — the last & highest-complexity breadth item; TIER B COMPLETE.** Unlike (1)–(3), which each added a data *source*, this changes the *query itself*, so it touches every literature/trial search. **Design choice surfaced to Victoria first** (like the intervention-type gate): how aggressive the synonym expansion should be — she chose **canonical + synonyms** (map the typed condition to its official MeSH heading AND search its entry-term synonyms), the recall/precision balance right for a regulator-grade tool, over "canonical only" (misses synonyms) or "+ hierarchy" (child terms → noise, hard to audit). **API research (de-risked live before coding — 6 probes):** the authoritative `id.nlm.nih.gov` descriptor lookup matches only *preferred* labels (so "heart attack" and the ambiguous "stress" both miss); its term lookup resolves a synonym but only to a term-id with no cheap path to the descriptor; the plain NCBI `esearch db=mesh` ranks garbage first ("stress"→Protein Aggregation, "melanoma"→Tetraspanin 30 — would misground). **The winner: NCBI E-utilities MeSH db with a FIELD-QUALIFIED `"{term}"[MeSH Terms]` query** — an exact controlled-vocabulary match that resolves synonyms cleanly ("heart attack"→Myocardial Infarction D009203, "psychological stress"→Stress, Psychological D013315) *and* correctly returns nothing for genuinely ambiguous tokens ("stress"/"MI"/"afib"), so it skips rather than mis-maps. `esummary` then yields `ds_meshui` (descriptor) + `ds_meshterms` (preferred first + synonyms). **`normalize_mesh(candidates)`** tries an ordered candidate list from **`_mesh_candidates`** (bigram-first, since MeSH headings are usually multi-word), first hit wins (deterministic), returns `{descriptor_id, preferred, synonyms, input}` or **None → raw keywords** (can only ADD precision, never make retrieval worse). Kept `build_queries` **pure** (no network) via an optional `mesh=` param; the lookup runs once in `build_grounded_context`, which re-calls `build_queries(mesh=…)` to rebuild the query as a **boolean OR-group** (canonical+synonyms) AND-ed with the distinctive method term — routed to the boolean-capable engines (CT.gov Essie, PubMed, Europe PMC) via a new `search_literature(bool_term=…)`, while OpenAlex/Semantic Scholar (relevance rankers) get a plain **bag** form. The resolved heading is logged in `### Retrieval Metadata` for audit. **Three bugs the end-to-end run caught that unit tests could not:** (a) **NCBI 429 throttling** — firing the candidate sweep back-to-back blew NCBI's un-keyed 3 req/s limit and 429'd *every* call → None; fixed with a rate-limited `_eutils_mesh_get` (≤3/s spacing + one retry on 429). (b) **Candidate flooding** — the flagship HRV case showed "none" because use_case bigrams ("continuous passive", "daily life") filled the candidate cap before the model's "psychological stress" was reached (and "daily life" risked mis-resolving to *Activities of Daily Living*); fixed by **condition-anchoring** the bigrams. (c) **Qualifier + terse-use_case** — "DR screening" has no condition word, so it anchored on "screening" which maps to the MeSH *qualifier* "diagnosis" (Q000175 — a subheading, never a condition); fixed by **rejecting non-D/C descriptors** AND anchoring on the model's clinical terms when the use_case is too terse, with the bare-token fallback kept narrow so generic model words ("lead"→Lead-the-metal, "skin"→Skin) can't mis-resolve. **Live-verified disease-agnostic across 6 indications:** melanoma→Melanoma, AFib→Atrial Fibrillation, sepsis→Sepsis, diabetic retinopathy→Diabetic Retinopathy, depression→Depression, pneumonia→Pneumonia; the golden HRV case now resolves "psychological stress"→Stress, Psychological (D013315)+4 synonyms and still returns 15 Crossref-verified articles.
