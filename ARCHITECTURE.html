Architecture — Clinical AI Eval Designer

How the app turns a clinical AI model + context into a grounded, citable 8-field validation specification. Companion to ARCHITECTURE.md (same version).

The three roles of Claude

Three distinct products get called "Claude." Keep them separate in the write-up.

WhereWhich ClaudeWhat it does
Discovery / R&DClaude Science
(workbench at claude.ai)		 Manual proof-of-concept: the 48-paper HRV synthesis that proved the concept. Not called by the app.
Building the appClaude Code Wrote app.py, the constraint-layer prompt, and the registry pipeline.
Runtime engineAnthropic API — Claude Fable 5 Synthesizes the retrieved records into the constrained 8-field spec.
Honest framing. The running app does not call Claude Science — there is no programmatic Claude Science endpoint. Claude Science was the discovery engine. The productized runtime is direct registry APIs + the Anthropic API (Fable 5). The registries — not the model's memory — are what make the citations real.

Runtime data pipeline

User input
  • AI model · clinical use case · patient population · healthcare setting
▼
1Fetch raw data — external registry APIs
  • ClinicalTrials.gov API v2 → study designs, endpoints, enrollment (NCT IDs)
  • openFDA device APIs → classification + 510(k)/De Novo (K-numbers, product codes)
  • PubMed E-utilities → literature (real PMIDs)
▼
2Assemble "Grounded Reference Context"
  • Raw records, clearly labelled by source, size-capped, injected into the prompt
▼
3Synthesize — Anthropic API, Claude Fable 5
  • Constraint layer: no invented numbers · cite-or-flag · confidence + expert-review tiers
  • Optional web_search fills registry gaps (e.g. FDA guidance PDFs)
  • Auto-fallback → Opus 4.8 on a safety false-positive
▼
Structured 8-field validation spec
  • + confidence flags · certifies / does-not-certify · real citations
Why the ordering matters. Registries are queried first, and their records are handed to Fable as source material. Fable's job is to map and synthesize real records into the 8-field structure — not to recall facts. An identifier it cites came back from a database this run, not from training memory.

Honesty guardrails

What the app deliberately does not claim:

  • Registry coverage is best-effort. A nil result is reported as "no matching record retrieved," and the model lowers confidence rather than asserting the benchmark doesn't exist.
  • Web search returns real pages but not structured database records — the model prefers registry identifiers and never fabricates one.
  • Every field carries a confidence tier and an expert-review flag. This is a grounded starting point, not a substitute for expert sign-off.

Roadmap

  • v3 (planned): ClinicalTrials.gov endpoint/statistical-plan extraction → auto-populate a Statistical Analysis Plan (SAP) skeleton.
  • v3+: IRB submission template and Q-Sub outline pre-populated from the generated spec — the workflow-integration moat.
